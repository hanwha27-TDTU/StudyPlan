# 정적 HTML 앱 점검 (static-html-review)

단일 파일 HTML 앱을 **보안을 제외하고** 전수점검한다. 목표는 "동작하지만 어긋나는 부분"을
우선순위로 짚어 주는 것. 막연한 칭찬·중복 나열을 피하고, `파일:줄번호`로 근거를 댄다.

## 진행 순서

1. **전체 읽기 먼저.** 파일이 크면 끝까지 읽고 나서 판단한다. 앞부분만 보고 단정하지 않는다.
2. JS 문법이 깨지지 않는지 빠르게 확인한다(예: `<script>` 본문을 `new Function(src)`로 파싱).
3. 아래 체크리스트로 훑고, **실제 버그 → UX 일관성 → 성능 → 유지보수** 순으로 정리한다.
4. 각 항목에 심각도(🔴/🟡/🟢)와 `줄번호`, 한 줄 수정 방향을 붙인다.
5. 끝에 "어디까지 고칠까요?"로 범위를 확인한 뒤에만 수정에 들어간다.

## 체크리스트

### 🔴 정확성 (가장 먼저)
- **상태 변경 후 렌더 누락**: 데이터를 바꾸는 핸들러가 관련 UI를 *모두* 다시 그리는가?
  (예: 체크 토글이 표만 갱신하고 상단 진행률·통계·차트는 안 그려 30초 뒤에야 반영되는 경우)
- **시간 기반 UI**: `setInterval` 주기와 계산 정밀도가 맞는가?
  (초 단위로 계산하면서 30초마다만 그리면 바늘이 점프한다)
- **경계/폴백 값**: "현재 항목 없음"일 때 엉뚱한 기본값(첫 항목 등)으로 폴백하지 않는가?
- **상태축 혼선**: 화면이 "지금(실시간)"과 "선택한 날짜/항목"을 섞어 보여 주지 않는가?

### 🟡 UX 일관성
- 모달이 ESC로 닫히는가? 바깥 클릭 정책과 키보드 정책이 의도대로인가?
- 헤더·라벨 문구가 실제 저장 단위(날짜별/모드별 등)와 일치하는가?
- 빈 상태(데이터 0건) 안내가 있는가?

### 🟢 성능 / 유지보수
- **렌더 루프 안에서 localStorage/JSON.parse를 반복**하지 않는가?
  렌더당 한 번 로드해 넘긴다.
- 함수 이름이 실제 동작과 맞는가? (예: prune라는데 아무것도 지우지 않음)
- 중복 함수·죽은 변수·stale 메타데이터(버전/날짜)를 정리한다.
- `escapeHTML` 같은 출력 이스케이프가 사용자 입력 렌더 전 구간에 일관 적용되는가?

## 산출물 형식

- 총평(좋은 점 1~2개 포함) → 🔴/🟡/🟢 순 목록 → 우선순위 한 줄 → "어디까지 고칠까요?".
- 추정·불확실한 항목은 그렇다고 표시한다. 확신 있는 버그와 섞지 않는다.

## 수정할 때
- 동작 동일성을 유지하고, 한 번에 하나의 의미 단위로 편집한다.
- 수정 후 다시 JS 파싱을 확인하고, 가능하면 헤드리스 브라우저로 렌더를 검증한다.
- 사용자가 요청한 범위에서만 commit/push 한다.
